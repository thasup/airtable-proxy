# Secure Airtable Proxy for AI Agents

A secure, high-performance proxy server built with FastAPI that sits between your AI agents and the Airtable API. 

When building AI agents (especially autonomous or third-party cloud agents like ChatGPT, LangChain, or Zapier), granting them direct access to your primary Airtable tokens poses a massive security risk. This proxy solves that by acting as a centralized gateway to manage authentication, enforce access scopes, and **completely hide your real Airtable Personal Access Tokens (PAT) from the public internet**.

---

## 🗝 The Three Keys (How it Works)

To understand this proxy, you need to understand the three different "keys" involved:

1. **`AIRTABLE_PAT` (The Master Key)**: This is your real Airtable token. **Only the proxy server knows this.** You never give this to an AI agent.
2. **`ADMIN_SECRET` (The Badge Maker)**: A password you make up. You use this to log into the proxy server to "print" new access badges for your AI agents. You **never** give this to an AI agent either.
3. **`sk_agent_...` (The Agent Badge)**: This is a custom access token generated by the proxy server. **This is what you give to your AI agent.** If the AI agent leaks this, you simply delete it from the proxy, keeping your real Airtable account safe.

## 🚀 Deployment (Render Free Tier)

This repository comes pre-configured for a frictionless deployment to [Render's Free Tier](https://render.com/).

> **Note for Free Tiers**: Render's free tier uses ephemeral storage. The `.db` file storing your generated `sk_agent_` tokens will wipe if the server sleeps due to inactivity. For production, upgrade to a paid tier or mount a persistent disk.

**Automated Deployment via `render.yaml`:**
1. Connect your Github/GitLab account to Render.
2. Under "Blueprints", click "New Blueprint Instance".
3. Select this repository.
4. Render will automatically parse the `render.yaml` file configuring the build.
5. In the Render Dashboard under your new Web Service, manually supply your secrets for `AIRTABLE_PAT` and `ADMIN_SECRET` in the **Environment Variables** tab. 

**Manual Local Setup:**
```bash
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
# Create a .env file with AIRTABLE_PAT and ADMIN_SECRET
./run.sh
```

## Step 1: Generate an `sk_agent_` Token 

Once your proxy is running (e.g., on `https://your-app-slug.onrender.com`), you need to use your `ADMIN_SECRET` to generate a token for your AI.

Run this command in your computer's terminal:

```bash
curl -X POST "https://your-app-slug.onrender.com/admin/tokens" \
     -H "Authorization: Bearer your_super_secret_admin_phrase" \
     -H "Content-Type: application/json" \
     -d '{
           "name": "My AI Assistant",
           "allowed_bases": ["*"]
         }'
```
*(Tip: `["*"]` allows access to all bases. You can restrict it by passing a specific base ID like `["appXYZ123"]`).*

**Response:**
```json
{
  "id": "b3e945...",
  "token": "sk_agent_AbCdEfGhIjKlMnOpQrStUvWxYz",
  "name": "My AI Assistant",
  "allowed_bases": ["*"],
  "created_at": "2026-03-11T00:00:00Z",
  "is_active": true
}
```

Copy the `token` value (`sk_agent_AbCd...`). This is the only time you will see it!

### Revoking a Token
If you need to cut off an AI agent's access, delete the token using its `id` from the creation step:
```bash
curl -X DELETE "https://your-app-slug.onrender.com/admin/tokens/b3e945..." \
     -H "Authorization: Bearer your_super_secret_admin_phrase"
```

## Step 2: Connecting your AI Agent 

Now that you have an `sk_agent_` token, you give **that token and the Render Proxy URL** to your AI agent, instead of the real Airtable API.

The proxy supports all normal Airtable endpoints (e.g. `GET /v0/baseId/TableId`). The proxy will intercept the request, verify the `sk_agent_` token, swap it for the real `AIRTABLE_PAT`, and forward the request to Airtable silently.

### Verifying it yourself via Terminal
To test if it works, run a normal Airtable request, but point it at your proxy and use your `sk_agent_` key:

```bash
curl -X GET "https://your-app-slug.onrender.com/v0/appXYZ123/Table1" \
     -H "Authorization: Bearer sk_agent_AbCdEfGhIjKlMnOpQrStUvWxYz"
```
You should see your raw Airtable data returned! Note: Pasting the URL in a web browser will result in `401 Unauthorized` because browsers do not send the secret `Authorization: Bearer` header.

### Integrating with Cloud AI (GPTs, LangChain, Zapier)
In your AI tool's API configuration settings:
1. **Base URL**: `https://your-app-slug.onrender.com`
2. **Authentication Type**: `Bearer Token` (or `API Key` in Header)
3. **Header Name**: `Authorization`
4. **Header Value**: `Bearer sk_agent_AbCdEfGhIjKlMnOpQrStUvWxYz`

## 🛡 Security & Specifications
- **Data Model:** Tokens and Access Logs are managed via a local `SQLite` file ensuring minimal external network dependencies.
- **Rate-Limiting:** Incorporates an in-memory sliding window rate-limiter rejecting excessive requests (`429 Too Many Requests`) to prevent brute force.
- **Highly Concurrent:** Powered by `FastAPI` and `httpx`, capable of handling hundreds of concurrent agent requests efficiently.
- **Audit Logging:** Every authenticated access attempt is catalogged (timestamp, success code, and requested path) alongside the unique agent ID utilizing background threads.
## 🛠 The "Render Free Tier" Database Catch

If you deploy this on Render's Free Web Service, you might notice your AI Agent suddenly loses access every few days (or hours if inactive).

**Why?** Render automatically shuts down free servers after 15 minutes of inactivity. When it wakes up from a new request, it boots a completely fresh container. This means the local `proxy.db` SQLite file where your `sk_agent_` tokens are saved is **wiped out**. 

To solve this for free, you have three options:
1. **The Automatic Scripting Approach:** Code your AI agent's startup sequence (or a scheduled script) to automatically call the `/admin/tokens` endpoint using your `ADMIN_SECRET` to fetch a new token whenever it gets a 401 Unauthorized error.
2. **The Render Background Cron Job:** Use a free Cron service (like cron-job.org) to ping your server's `https://your-app-slug.onrender.com/health` endpoint every 10 minutes to prevent it from ever going to sleep.
3. **Upgrade:** The $7/month Render tier prevents sleeping, but you will still need to attach a "Render Disk" to persist the `.db` file across manual deployments.
